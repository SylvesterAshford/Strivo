# Design: Advisor Home V1 — Monthly Profit Review

Generated by /office-hours · Strivo (Burmese-first MSME finance) · Status: APPROVED · Mode: Startup (pilot/PMF)
Refines: `PLAN.md`, `strivo-advisor-home-implementation-plan.md`

## The one decision that reframed everything

Pilot owners **have daily data but will only enter it weekly or monthly** (most likely
monthly). That is fixed behavior, not a habit to fix. So:

- A "Today's sales" Home is the wrong product for them — it shows "no sales today"
  almost always and looks broken. (This also means the daily dashboard already built
  in `src/app/(tabs)/page.tsx` must be reframed.)
- The Advisor Home plan's "last 7 days / today" windows are wrong for batch input.

**Strivo V1 is a periodic profit review, not a daily app.** Built on a **monthly**
clock (chosen over weekly/auto to keep one simple rhythm and avoid empty states).

## Problem statement

A Myanmar shop owner can't easily answer, for the period they actually track:
*did I make money, what changed, and what should I do next* — without accounting
skills, and without entering data daily. Existing tools assume daily use and
double-entry. Strivo's edge is AI batch extraction (paste/Excel) + plain-Burmese advice.

## Demand evidence

**None yet. Pre-product, zero users.** This plan is a hypothesis. The first job is to
test it with real owners, not to build more (see The Assignment).

## Status quo (the real competitor)

The owner's gut + the cash drawer at month end. A "Watch / At Risk" status must be
*more trusted than their own gut*, and one wrong status (from stale/incomplete data)
burns that trust permanently. Mitigation: staleness gating + deterministic rules.

## Target user & narrowest wedge

- **User:** owner of a small retail / F&B / services shop who keeps records but enters
  them into Strivo in monthly batches.
- **Wedge (the one thing to test this month):** *"Paste or upload this month's sales
  and expenses, get a clear profit review + one action."* If owners won't do that, no
  dashboard polish saves the product.

## The core loop

```
1. Remind   →  prominent "Update this month" CTA when data is stale   (push/notify deferred)
2. Batch in →  paste-text (AI extract) or Excel upload                ← magic moment, target < 2 min
3. Review   →  monthly profit · status word · why it changed vs last month · ≤3 actions
4. Did it?  →  done / skip on each action                             ← the instrument (in V1)
```

## Premises (agreed this session)

1. The pilot constraint is **fitting a monthly batch rhythm**, not forcing daily input. **Agree.**
2. The **action loop ships in V1**, not a later phase — it's the only measure of whether
   advice is useful. **Agree.**
3. **Defer** the 0-100 score number and goal-capture; status word + diagnosis is enough. **Agree.**
4. This is a **hypothesis needing live owners** before building more. **Agree.**

Kept from the original plans (do not change): deterministic-first advice, confidence/
staleness gating, ≤3 actions, "understandable in 5 seconds," and the explicit not-yet
list (no multi-agent, benchmarks, inventory, product profitability, forecasts).

## Approaches considered

- **A. Minimal monthly review + action loop (CHOSEN).** Reframe Home to monthly; ship
  import-first CTA when stale, monthly profit snapshot, status word (no number),
  why-it-changed (month vs last month), ≤3 deterministic actions with done/skip. Reuses
  the existing import flow, `home` route, and the cards already built. Smallest thing
  that tests "monthly drop + act + return."
- **B. Full Advisor Home V1 (monthly).** All five sections incl. internal score, action
  loop deferred. More surface, learns less per unit of build, missing the act signal.
- **C. "Close the month" ritual-first.** Guided import→confirm→review ceremony. Best
  behavior-fit, most new UX before the first user test.

## Recommended approach: A

Tests the real hypothesis with the least build and includes the action loop. B and C
build more before learning; revisit C once owners prove they'll do the monthly drop.

## V1 scope — what to build

### Home (monthly), reframed from the daily dashboard
- **Period header:** "This month · through {date}". If the latest fact is old, show
  staleness: "Your data is from {month}. Add this month to see your review."
- **Update CTA** (primary) → routes to the existing import hub (`/record`).
- **Profit snapshot:** month sales, expenses, **profit** (profit is the hero figure).
- **Status word:** Good / Watch / At Risk (deterministic; no number shown in V1).
- **Why it changed:** this month vs last month — the single clearest deterministic
  reason (profit up/down because sales moved / expenses moved / both).
- **Actions (≤3):** deterministic, each with **Done / Skip**.
- **Recent entries** at the bottom (confirm the data landed).

### Deterministic rules (monthly windows)
- Status: start neutral; +profit positive, −profit negative, −expense ratio > 70%,
  −sales fell vs last month, −no data this month, +sales grew. (Internal score may
  exist but is not displayed.) Good ≥ 75 · Watch 45-74 · At Risk < 45.
- Why-it-changed: compare current month vs previous month profit/sales/expenses; pick
  the dominant mover. Never claim causes the data can't support (no "repeat customers",
  "inventory", "cash runway").
- Alerts (≤2 in V1): expense ratio > 70%; outstanding receivables exist.
- Actions: rule-based (review biggest expense category; follow up unpaid money; record
  this month if stale; keep recording if stable).

### Confidence / staleness
- `insufficient`: < 3 facts or no sales in the current+previous month → honest "add data".
- `partial`: some data but thin → cautious advice.
- `enough`: a full month of data present.
- Always surface "data through {date}" so advice is never mistaken for live.

### Action loop (in V1)
- Each action supports **done / skip** (drop "planned/not-useful" for V1 simplicity).
- Persist `advisor_action_events { id, workspaceId, actionKey, status, createdAt }`.
- This is the primary success signal — instrument it from day one.

## What changes in the existing code (no implementation now)

- New: `advisor_action_events` table + a confirm endpoint for done/skip.
- `src/lib/api.ts` — add the `AdvisorHome` types (monthly).

## Deferred (NOT in V1)

0-100 score number · goal capture · weekly/auto rhythm · push reminders · benchmarks ·
product profitability · forecasts · inventory · multi-agent. Reminders matter but can
start as an in-app stale banner before any notification system.

## Open questions

- How will owners be reminded to do the monthly drop without push notifications? (in-app
  banner is the V1 answer; revisit SMS/Viber later — that's the real Myanmar channel.)
- Does "month" mean calendar month or last-30-days for the review? (calendar month is
  simpler to explain to owners; confirm during field test.)

## Success criteria (leading indicators only)

- An owner completes a monthly data drop (the magic moment).
- They return the next month and do it again.
- They mark at least one action **done**.

Do NOT optimize for charts, AI pages, prompts, or "owner-reported profit improvement"
(unattributable at pilot).

## Eng Review Decisions (locked)

1. **Structure (minimal):** one pure `src/lib/advisor/monthly.ts`, computed inside
   `home/route.ts` (no separate GET). Action loop = 1 table + 1 POST endpoint.
   Frontend = reframe `(tabs)/page.tsx` + 2 components (AdvisorCard, AdvisorActions).
   Reuse the calendar-month query pattern from `reports/route.ts` and the existing
   card system. Do NOT extend `computeMetrics` (keep the insights engine's
   rolling/half-vs-half semantics intact).
2. **Isolation (A1):** `advisor_action_events` gets RLS mirroring `facts` —
   `USING (workspace_id = current_setting('app.workspace_id'))`, written inside
   `withMobileAuth`. No app-layer-only filtering.
3. **Period (A2):** Home headline = **the most recent month that has data**, compared
   to the month before it. Right after a catch-up it shows the month just entered, never
   an empty current month. One month only → "first month, keep recording" (no
   comparison). Staleness line: "data through {date}".
4. **DRY date math (CQ1):** month-window helpers (`monthBounds`, `latestMonthWith`,
   `priorMonth`) live in `monthly.ts`; the route calls them. No third inline copy.
5. **Action keys (CQ2):** stable `ActionKey` enum (`review_top_expense`,
   `follow_up_receivables`, `record_more`, `keep_recording`, …) separate from the
   Burmese display title. Events store the key. The action object is
   `{ key, title, reason, priority }`.
6. **Failure mode (3A):** wrap `buildAdvisor` in try/catch in the route; on throw, log
   and return `advisor: null` with the rest of `/home` intact. Frontend hides the
   advisor card. Derived-logic bug never blocks Home. (Invariant: derived data degrades,
   fetched data surfaces errors.)
7. **Inputs (4A):** the route runs SQL aggregates (this/last month sales+expense,
   outstanding, top expense category, latest-month-with-data) in the existing
   `Promise.all`; `buildAdvisor` is pure over plain numbers. No fact rows loaded in JS.

## Data flow

```
GET /api/mobile/v1/home  (withMobileAuth → RLS app.workspace_id set)
  └─ Promise.all([ ...existing home sums,
                   latestMonthWith(facts),           ← max(occurredAt) bucketed
                   sum(sales)  this-month, last-month,
                   sum(expense) this-month, last-month,
                   sum(receivable) outstanding,
                   top expense category this-month ])
       └─ try { advisor = buildAdvisor(aggregates) }   ← pure, src/lib/advisor/monthly.ts
          catch { advisor = null }                       ← degrade
  → { ...homeFields, advisor }

POST /api/mobile/v1/advisor/action  (withMobileAuth → RLS)
  body { actionKey: ActionKey, status: 'done'|'skip' }  ← Zod-validated
  → insert advisor_action_events (workspace-scoped)
```

## Failure modes

| Codepath | Failure | Test? | Error handling | User sees |
|---|---|---|---|---|
| `buildAdvisor` derivation | throws on odd data | unit (force throw) | try/catch → `advisor:null` | advisor card hidden; basic numbers OK |
| diagnosis | first month, no prior | unit | first-month branch | "first month, keep recording" |
| `priorMonth` | year boundary Dec→Jan | unit | explicit date math | correct prior month |
| action POST | cross-workspace write | **deferred E2E** | RLS denies | write blocked (untested in V1) |
| `/home` | real DB error | (existing) | throws → QueryError | retry screen |

**Critical gap:** action-endpoint RLS isolation has **no automated test in V1** (no
integration harness yet). RLS denies at the DB, so it's not a silent failure, but it's
unverified — folds into the existing E2E TODO from the last review.
